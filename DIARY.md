# Дневник проекта PHF10 Evolution

## Этап: интеграция AIUPred в пайплайн анализа IDR

Сегодня был реализован первый полноценный модуль анализа IDR на основе AIUPred.

### Исходная задача

Цель состояла в том, чтобы получать для PHF10:

* вероятность беспорядка (disorder);
* вероятность участия в связывании (binding);

и сохранять результаты в стандартизированном формате для дальнейшего анализа.

В качестве тестового белка использовался PHF10-Pl (UniProt: Q8WUB8).

---

## Получение данных из AIUPred

Вместо веб-интерфейса был использован REST API AIUPred.

Реализована функция:

```python
get_aiupred(accession)
```

которая обращается к:

```text
https://aiupred.elte.hu/rest_api
```

и получает JSON с результатами.

Для PHF10 API возвращает:

```text
accession
options
AIUPred
AIUPred-binding
```

где:

* AIUPred — значения disorder score;
* AIUPred-binding — значения binding score.

Было установлено, что обе структуры представляют собой массивы чисел длиной, равной длине белка.

Пример первых значений:

```text
Disorder:
0.9725
0.9734
0.9740
...

Binding:
0.9639
0.6243
0.5861
...
```


Содержимое:

Хранение её в одном месте уменьшает дублирование кода.

## Переход на пакетный запуск

Первоначально скрипты запускались напрямую:

```bash
python src/idr/run_aiupred.py
```

Это приводило к проблемам с импортами.

Был осуществлён переход на запуск модулей через пакетный механизм Python:

```bash
python -m src.idr.run_aiupred
```

Для этого был добавлен файл:

```text
src/__init__.py
```

а также:

```text
src/utils/__init__.py
```

---

## Почему выбран запуск через -m

При запуске:

```bash
python -m src.idr.run_aiupred
```

Python воспринимает весь проект как пакет:

```text
src
 ├── idr
 └── utils
```

Благодаря этому становятся возможны стабильные импорты:

```python
from src.utils.fasta import read_fasta
```

Такой подход особенно важен для крупных проектов, где модули активно взаимодействуют друг с другом.

Он устраняет необходимость в:

```python
import sys
sys.path.append(...)
```

и делает структуру проекта воспроизводимой на других машинах.

# Length check
Получено 211 ортологов PHF10 из Ensembl Compara и сформирован единый FASTA-набор. Проведён QC по длинам последовательностей.

Диапазон длин: 228–6756 а.о., средняя длина ~520 а.о. Распределение сильно неоднородное, нормальное приближение не подходит (mean ± 2σ даёт некорректные границы).

Выполнена первичная кластеризация длин методом Gaussian Mixture Model с выбором числа компонент по BIC (без фиксированных порогов). Зафиксирована необходимость рассматривать распределение длин как смесь нескольких подгрупп, а не как единый класс.

Вывод: набор ортологов содержит структурно разнородные последовательности, требующие дальнейшей стратификации перед MSA и филогенетическим анализом.

Зафиксировано важное методологическое уточнение после первого strict MSA и alignment QC.

Текущий набор `msa_input_strict` нельзя считать финально строгим биологически. Он является технически строгим набором: последовательности прошли первичный length/domain/manual QC, были использованы для MAFFT, а выравнивание прошло технический alignment QC.

Результаты текущего alignment QC:

* последовательностей: 198;
* длина alignment: 1353;
* несоответствий между исходной длиной белков и non-gap длиной не обнаружено.

Однако текущий QC-слой не выявляет все биологически важные проблемы. GMM/length QC ловит в основном экстремальные выбросы по длине, но может пропускать:

* обрезанные N-концы;
* фрагменты средней длины;
* неполные транскрипты;
* ошибки gene prediction;
* проблемные белковые модели;
* множественные записи на один вид;
* X-символы в критических областях.

Принято решение не считать отсутствие N-концевого участка эволюционной потерей до исключения технических причин: неполных транскриптов, ошибок аннотации и обрезанных моделей.

Также зафиксировано, что `TRUNCATED` не равно `JUNK`. Обрезанные последовательности могут быть реальными ортологами и быть пригодными для общего дерева, но непригодными для анализа N-концевого IDR и фосфо-кластера.

Дивергентные N-концевые участки у рыб, амфибий и других удалённых таксонов не должны автоматически классифицироваться как мусор на основании слабого совпадения с человеческим Ser-rich мотивом. Это может быть биологически значимый сигнал, а не ошибка.

Принято архитектурное решение: дальше нужен не ещё один фильтр ортологии, а отдельный слой sequence/readiness QC.

Новый слой должен классифицировать пригодность каждой последовательности отдельно для разных задач:

* tree analysis;
* N-terminal analysis;
* PTM / phospho-cluster analysis.

Один общий набор больше не рассматривается как достаточный для всех downstream-анализов.

Следующий этап:

#sequence_readiness_qc

Реализовать модуль `src/orthology/sequence_readiness_qc.py`, который будет оценивать:

* покрытие N-концевого окна относительно человеческой последовательности;
* пригодность записи для N-terminal анализа;
* пригодность записи для PTM анализа;
* X-символы в полной последовательности и в N-terminal/PTM окне;
* множественные записи на один вид;
* отдельные readiness-статусы для дерева, N-конца и PTM-анализа.

Модуль не должен переопределять ортологию, фильтровать последовательности или использовать Ser-rich мотив, N-terminal extension или фосфосайты как критерии ортологии.
