# Дневник проекта PHF10 Evolution

## Этап: интеграция AIUPred в пайплайн анализа IDR

Сегодня был реализован первый полноценный модуль анализа IDR на основе AIUPred.

### Исходная задача

Цель состояла в том, чтобы получать для PHF10:

* вероятность беспорядка (disorder);
* вероятность участия в связывании (binding);

и сохранять результаты в стандартизированном формате для дальнейшего анализа.

В качестве тестового белка использовался PHF10-Pl (UniProt: Q8WUB8).

---

## Получение данных из AIUPred

Вместо веб-интерфейса был использован REST API AIUPred.

Реализована функция:

```python
get_aiupred(accession)
```

которая обращается к:

```text
https://aiupred.elte.hu/rest_api
```

и получает JSON с результатами.

Для PHF10 API возвращает:

```text
accession
options
AIUPred
AIUPred-binding
```

где:

* AIUPred — значения disorder score;
* AIUPred-binding — значения binding score.

Было установлено, что обе структуры представляют собой массивы чисел длиной, равной длине белка.

Пример первых значений:

```text
Disorder:
0.9725
0.9734
0.9740
...

Binding:
0.9639
0.6243
0.5861
...
```

## Рефакторинг структуры проекта

В процессе работы проект был переведён от набора независимых скриптов к модульной структуре Python.

Текущая структура:

```text
src/
├── idr/
├── phosphosites/
├── motifs/
├── evolutionary_features/
├── figures/
└── utils/
```

Ранее использовались директории вида:

```text

```

Однако для Python-модулей это неудобно, поскольку идентификаторы не должны начинаться с цифр.

Поэтому директории были переименованы в валидные имена модулей.

---

## Создание пакета utils

Функция чтения FASTA была вынесена в отдельный модуль:

```text
src/utils/fasta.py
```

Содержимое:

```python
def read_fasta(path):
    ...
```

Причина:

одна и та же функция понадобится многим этапам проекта:

* AIUPred;
* flDPnn;
* консенсус IDR;
* построение MSA;
* работа с ортологами.

Хранение её в одном месте уменьшает дублирование кода.

---

## Переход на пакетный запуск

Первоначально скрипты запускались напрямую:

```bash
python src/idr/run_aiupred.py
```

Это приводило к проблемам с импортами.

Был осуществлён переход на запуск модулей через пакетный механизм Python:

```bash
python -m src.idr.run_aiupred
```

Для этого был добавлен файл:

```text
src/__init__.py
```

а также:

```text
src/utils/__init__.py
```

---

## Почему выбран запуск через -m

При запуске:

```bash
python -m src.idr.run_aiupred
```

Python воспринимает весь проект как пакет:

```text
src
 ├── idr
 └── utils
```

Благодаря этому становятся возможны стабильные импорты:

```python
from src.utils.fasta import read_fasta
```

Такой подход особенно важен для крупных проектов, где модули активно взаимодействуют друг с другом.

Он устраняет необходимость в:

```python
import sys
sys.path.append(...)
```

и делает структуру проекта воспроизводимой на других машинах.

# Length check
Получено 211 ортологов PHF10 из Ensembl Compara и сформирован единый FASTA-набор. Проведён QC по длинам последовательностей.

Диапазон длин: 228–6756 а.о., средняя длина ~520 а.о. Распределение сильно неоднородное, нормальное приближение не подходит (mean ± 2σ даёт некорректные границы).

Выполнена первичная кластеризация длин методом Gaussian Mixture Model с выбором числа компонент по BIC (без фиксированных порогов). Зафиксирована необходимость рассматривать распределение длин как смесь нескольких подгрупп, а не как единый класс.

Вывод: набор ортологов содержит структурно разнородные последовательности, требующие дальнейшей стратификации перед MSA и филогенетическим анализом.

# requierments
1. Исходное состояние

В проекте существовал единый requirements.txt, содержащий смешанный стек:

научные библиотеки (numpy, scipy, pandas)
биоинформатика (biopython, metapredict, AIUPred и др.)
ML / deep learning стек (torch, pytorch-lightning)
CUDA / nvidia зависимости

Проблема: отсутствовало разделение по назначению зависимостей, окружение было неструктурированным.

2. Попытка стабилизации окружения

Было принято решение:

сохранить текущее окружение как snapshot состояния системы
разделить зависимости по функциональному назначению
3. Разделение зависимостей

Созданы два логических слоя:

3.1. Core dependencies (рабочий слой проекта)

Минимальный набор библиотек для PHF10 pipeline:

numpy
scipy
pandas
scikit-learn
requests
biopython
tqdm
matplotlib

Назначение: выполнение ортологии, QC, статистического анализа и базовой визуализации.

3.2. Snapshot окружения

Создан файл:

requirements_full_snapshot.txt

Назначение:

полная фиксация текущего состояния Python окружения
включает ML / CUDA / bio tools
используется только для воспроизводимости, не как рабочая конфигурация
4. Изменение структуры проекта
requirements.txt удалён или выведен из активного использования
введён принцип разделения:
core (минимальный набор)
snapshot (полный слепок состояния)
5. Работа с Git и окружением
исключены виртуальные окружения из Git (env/, env_clean/)
предотвращено попадание site-packages в репозиторий
восстановлен контроль над staging area после случайного добавления env
6. Текущее состояние системы
одно активное Python-окружение используется для разработки
зависимости разделены логически
репозиторий содержит только код и артефакты анализа
окружение зафиксировано через snapshot
7. Итог

Система приведена к структуре:

код отделён от окружения
зависимости разделены на рабочие и архивные
Git больше не отслеживает виртуальные окружения
проект готов к дальнейшему развитию ортологического и эволюционного анализа
